26 //-------------------------------------------------------------------//
27 // How to use //
28 //-------------------------------------------------------------------//
29 /*
30 * 1. Get the ECSManager instance by calling ECSManager::GetInstanceHandle();
31 * Save this reference somewhere, maybe ECSManager& ecs = ECSManager::GetInstanceHandle();
32 * Note: This call may fail if you're getting the manager instance before it is created by the Engine.
33 * Only call this function AFTER the engine has added this manager to itself.
34 *
35 * 2. Create an EntityRef and assign an entity to it. Refer to class summary for more info.
36 * There are 3 ways to assign: CreateEntity, FindEntity, CloneEntity (explained later)
37 * E.g.
38 * EntityRef newEntity = CreateEntity();
39 * EntityRef newEntity = CreateEntity(someGUID); // only used when deserializing an entity that already has a GUID. Leave empty otherwise.
40 * EntityRef existingEntity = FindEntity("Name of entity");
41 * EntityRef existingEntity = FindEntity(someGUID);
42 *
43 * 3. Add/remove/check/get components from entities.
44 * newEntity.AddComponent<Transform>();
45 * newEntity.AddComponentGeneric(typeGUID); // only used when deserializing. tpyeGUID is the component's typeguid.
46 *
47 * newEntity.RemoveComponent<Transform>();
48 * bool hasTransform = newEntity.CheckComponent<Transform>();
49 * Transform& tComp = newEntity.GetComponent<Transform>(); // will throw error if entity does not have component. Use CheckComponent prior to avoid error.
50 * std::vector<IComponent*> components = newEntity.GetAllComponents();
51 * // can loop thru components[i] to see what components this entity has.
52 * // can access generic functions in IComponent to serialize, deserailize, get GUID or typeGUID.
53 *
54 * 4. Cloning/copying entities
55 * EntitiyRef ent1Clone = ecs.CloneEntity(ent1); // ecs1Clone will be newly created with an identical copy of ent1's components and data.
56 * // However, the GUID of ent1Cone is NOT the same as ent1. Some other unique data won't be copied as well.
57 * ecs.CopyOverwriteEntity(ent1, ent2); // ent2 will have all its data overwritten and replaced with data from ent1. All its components will be added/removed
58 * // until it's the same as ent1.
59 * // However, the GUID of ent1Cone is NOT the same as ent1. Some other unique data won't be copied as well.
60 * 5. Destroy entity
61 * ecs.DestroyEntity(newEntity); // destroys this entity and sets EntityRef to null. Does not affect other EntityRefs referencing this entity (not good)
62 * ecs.DestroyAllGameEntities(); // destroys all in-game entities. Does not destroy guys flagged as "dont destroy on load" or "prefabs". Can enable bools to destroy these too.
63 *
64 * ================= For ECS system writing =================
65 * 1. Getting component arrays to loop through
66 * if (!ecs.CheckComponentRegistered<Transform>()) return;
67 * auto& compArray = ecs.GetComponentArray<Transform>();
68 * for (auto& component : compArray) { do some processing for each component }
69 *
70 * 2. Finding out which entity this component belongs to:
71 * for (auto& component : compArray) {
72 * EntityRef ent = component.GetEntity(); // inherited from IComponent, so IComponents can do this too.
73 * ent.GetComponent<Rigidbody>().SetVelocity(xxx); // getting another component of this same entity
74 * }
75 */

920// NOTES:
921// EntityGUIDToECSIndex:
922// will help bring the GUID level down to ECSIndex level.
923// ECSIndex level is volatile, the same ECSIndex cannot be guranteed across different scenes for the same entity.
924// Entity's GUID will persist across loads, so use that instead to identify Entities when loading in or saving entities.
925//
926// EntityIndexFreeList:
927// Helps to track which ECSIndexes are open to use. If an index is taken, pop it off the list.
928// If an entity is destroyed, put its index back on the free list.
929//
930// EntityStorage:
931// Vector of _EntityData. Each _EntityData is identified by the vector's index. ECSIndex is also
932// based off this vector. So ECSIndex of 0 will refer to m_EntityStorage[0], 1 to m_EntityStorage[1] etc.
933//
934// ComponentStorage:
935// Stores vector of different types of Component Containers, hence std::any is used to make these types generic
936// to store them together. ComponentContainer is identified by its TypeGUID (IObject interface). To find the
937// desired ComponentContainer for desired type, get the TypeGUID of this type. Then search through this vector for
938// the ComponentContainer that matches the TypeGUID. This result of searching can be cached in a template function,
939// so subsequent calls will avoid searching again.
940//
941// ComponentContainer:
942// A templated class that contains the array of components. One instance of this class will be generated for each component type.
943// ComponentContainer uses sparse and dense array concept to access components very quick.
944//
945// SystemStorage:
946// Works on the exact same principle as ComponentStorage, just with Systems now. Refer above.
947//
948// Special note:
949// Try to make SCRIPTS work with this ECS system.
950// PROBLEM:
951// I can only have 1 entity per Component. SO only 1 script component per entity
952// I should make one new ComponentContainer for every new script class.
953// However, I cannot know the script Classes during compile time. I can only know them during compile time.
954// So creating unique ComponentContainers via templates is out of the window.
955// Possible solutions:
956// 1. Create a ComponentContainer but specialize it to contain script identification.
957// Downside: Locating each ComponentContainer is going to be expensive as i cannot generate template funcs to cache values.
958// 2. make a new ScriptStorage, this will make use of a MAP to locate script containers. Will require a custom ScriptContainer too.
959// Downside: Map slow. But i guess this is the most viable option over searching thru all containers linearly.
960// Fix: Make ScriptID be the order i read off custom classes, and the order that these ScriptContainers are stored.
961// Then i can access them using a direct array ref.
962// Issue is, will i be able to tell the MonoScript side to use this ScriptID? idk
